---
title: "令牌刷新"
description: "@ag-vibe/auth 如何处理访问令牌的主动刷新与被动刷新。"
---

auth 客户端在两种情况下刷新访问令牌：令牌即将过期时主动刷新，以及请求返回 401 时被动刷新。

## 主动刷新

每次调用 `ensureValidAccessToken()` 时，客户端会检查当前令牌是否将在 60 秒内过期。如果是，则在返回令牌之前先调用 `POST /auth/refresh`。

这意味着你的 API 请求几乎不会携带过期令牌到达服务端——刷新在后台提前完成。

你可以在直接使用底层 JWT 工具时自定义提前量：

```ts
import { isTokenExpiringSoon } from '@ag-vibe/auth'

// 使用 5 分钟提前量，而不是默认的 1 分钟
isTokenExpiringSoon(accessToken, 5 * 60 * 1000)
```

## 被动刷新（401 恢复）

安装 `applyTo` 拦截器后，任意 API 请求返回 401 都会触发强制刷新：

1. 拦截器调用 `ensureValidAccessToken(true)`——绕过过期检查，直接请求 `/auth/refresh`。
2. 刷新成功后，用新的 `Authorization` 头重试原始请求一次。
3. 刷新失败时，将原始 401 响应返回给调用方，不做额外处理。

认证端点（`/auth/sign-in`、`/auth/sign-up`、`/auth/sign-out`、`/auth/refresh`）被排除在重试逻辑之外，以防止无限循环。

已重试过的请求会带有内部标记头 `x-ag-vibe-auth-retried`，第二次收到 401 时直接透传。

## 并发去重

若多个请求同时收到 401，只会发出一次刷新请求。所有并发调用方共享同一个 in-flight Promise，并获得相同的结果。

## 失败处理

并非所有刷新失败都等同处理：

| 刷新响应 | 行为 |
| --- | --- |
| `401` 或 `403` | 清除会话——刷新令牌已失效。 |
| 网络错误、`429`、`5xx` | 保留会话——属于瞬时故障，用户维持登录状态。 |

这一区别对网络不稳定的用户尤为重要：刷新请求超时不会导致用户被登出。
